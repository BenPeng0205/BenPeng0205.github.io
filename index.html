别把报警写成一堆 IF：CodeSys 报警程序框架应该先解决什么

CodeSys 报警程序框架刊物级文章发布前预览稿。

CodeSysPLC报警框架ControlRookie

摘要：很多 PLC 报警程序最后会变成一片零散的 IF、R_TRIG 和 HMI 变量。刚开始能用，现场一复杂就开始失控：报警重复跳、确认逻辑不一致、历史记录缺字段、HMI 显示和程序状态对不上。本文先不急着给完整源码，而是把 CodeSys 报警程序框架拆成一套可复用的工程模型：检测、锁存、确认、抑制、恢复、记录、展示和排障。后续如果结合你的实际 ST 工程，可以继续落到真实功能块和源码逐段讲解。

适合谁收藏

  • 正在用 CodeSys / IEC 61131-3 写设备报警、产线报警或工艺异常处理的工程师。
  • 已经写过报警点，但发现确认、复位、历史记录和 HMI 联动越来越难维护的人。
  • 想把“能跑的报警逻辑”整理成“可扩展、可排障、可交付”的程序框架的人。
  • 准备把报警程序做成系列教程、组件库或项目标准模板的人。

先给结论

一个可靠的报警框架，核心不是“发现异常就置位一个报警位”，而是要回答六个问题：

1. 异常怎么进入系统：报警条件来自采集值、状态机、通信诊断，还是外部设备返回码。 2. 报警怎么保持：瞬时异常是否锁存，恢复后是否仍需人工确认。 3. 谁能改变状态：程序自动恢复、操作员确认、维护人员屏蔽，边界必须分清。 4. HMI 看到什么：当前报警、未确认报警、历史报警不能混成一个变量。 5. 历史怎么追溯：时间、来源、等级、确认人、恢复时间、上下文值必须结构化。 6. 现场怎么排障：报警不是提示语，它应该把工程师带到正确的检查路径。

如果这六个问题没有先定下来，后面写多少功能块都只是补丁。

现场最容易踩的坑

报警程序最常见的退化路径，是从一个看起来很合理的写法开始的：

IF xPressureHigh THEN
    xAlarmPressureHigh := TRUE;
END_IF;

IF xAck THEN
    xAlarmPressureHigh := FALSE;
END_IF;

这段逻辑在调试台上可能没有问题，但一到真实设备就会暴露边界缺口。

压力已经恢复，但报警是否应该自动消失？ 操作员点了确认，但异常条件还在，报警是否允许清掉？ 报警重复触发，历史记录是新增一条，还是更新同一条？ 维护模式下屏蔽报警，恢复生产后是否要重新评估？ HMI 显示“已确认”，程序内部到底是 Acked、Recovered，还是 Inactive？

这些问题不是 UI 问题，也不是变量命名问题，本质是报警状态模型不完整。

一套更稳的报警状态模型

我更建议把报警点抽象成一个小型状态机，而不是一个布尔量。

状态含义进入条件离开条件
Normal无报警条件未触发检测到报警条件
ActiveUnacked报警触发，未确认条件触发操作员确认或条件恢复
ActiveAcked报警仍存在，已确认报警触发后收到确认条件恢复
RecoveredUnacked条件已恢复，但未确认条件恢复但未确认操作员确认
Suppressed被抑制或屏蔽维护/联锁/工艺阶段不允许报警抑制条件解除

这里有一个关键判断：确认不是复位，恢复也不是确认。

确认代表“人已经看到了”；恢复代表“异常条件已经消失”。这两个动作来自不同主体，不能用一个按钮或一个变量混过去。

程序结构应该怎么拆

一个可复用的 CodeSys 报警框架，至少应该分成六层。

1. 报警条件检测层

这一层只回答一个问题：当前是否满足报警条件。

例如压力高、温度低、变频器故障、通信超时、阀门反馈不到位，都应该先被整理成统一的布尔触发条件和上下文值。

这一层不负责确认，不负责历史记录，也不直接写 HMI 文本。它只产出事实：

  • 当前值是多少。
  • 阈值是多少。
  • 条件是否满足。
  • 是否需要延时滤波。
  • 来源对象是谁。

2. 延时与去抖层

现场信号不干净，很多报警不能一触发就进入系统。压力波动、通信瞬断、传感器抖动，都需要延时确认。

但延时逻辑也不能散落在每个报警点里。否则几十个报警点以后，你会发现每个点的延时、复位和边界都不一样。

建议统一抽象为：

fbDelayOn(IN := xRawAlarm, PT := tOnDelay);
xAlarmCondition := fbDelayOn.Q;

这只是示意，不是你的工程源码。真正发布源码教程时，必须从实际 .st 文件逐字摘录，不能临时重写。

3. 报警状态机层

这一层是报警框架的核心。

它接收 xAlarmCondition、xAck、xSuppress、xReset 等输入，输出当前状态、是否需要显示、是否需要记录、是否需要通知。

状态机的好处是边界明确：

  • 条件还在时，确认只能从 ActiveUnacked 进入 ActiveAcked，不能直接清除。
  • 条件恢复但未确认时，进入 RecoveredUnacked，提醒操作员这件事发生过。
  • 抑制状态只影响报警呈现，不应该无痕吞掉全部历史。
  • 同一报警点重复触发时，要有明确的重新进入规则。

4. 报警目录与元数据层

报警不是只有一个变量名。一个正式项目里，每个报警点至少需要这些元数据：

字段作用
AlarmId稳定唯一标识，用于历史记录和 HMI 映射
Source设备、功能块、轴、工艺段或通信对象
Severity提示、警告、故障、停机等级
Message给操作员看的简短文本
Cause可能原因
Action首选检查路径
ContextValue触发时的关键过程值

很多报警系统不好用，不是因为检测不准，而是因为提示语没有工程指向。 “压力报警”价值很低，“A 缸推进压力超过上限，先检查比例阀输出、压力传感器和回油通道”才有排障价值。

5. 历史记录层

历史记录不能只保存一行字符串。

至少要区分：

  • 触发时间。
  • 恢复时间。
  • 确认时间。
  • 当前状态。
  • 报警等级。
  • 报警来源。
  • 触发上下文值。
  • 操作员确认来源。

如果历史记录没有这些字段，后期做追溯、复盘、质量分析和停机统计都会很痛苦。

6. HMI 展示层

HMI 不应该直接绑定一堆内部报警变量。更稳的做法是让程序提供已经整理好的视图：

  • 当前活动报警列表。
  • 未确认报警列表。
  • 历史报警列表。
  • 报警详情。
  • 过滤条件：等级、设备、时间、状态。

这样 HMI 只负责展示和交互，报警框架负责状态一致性。

一个推荐的对象拆分

如果后续要落到 CodeSys 工程，我会建议按这个方向拆：

对象职责
DUT_AlarmState定义报警状态枚举
DUT_AlarmConfig定义报警点配置和元数据
DUT_AlarmRuntime定义运行时状态、时间戳、上下文值
FB_AlarmPoint单个报警点的状态机
FB_AlarmManager管理报警点集合、列表输出、历史写入
GVL_AlarmConfig全局报警目录
PRG_AlarmTask周期调用入口，与设备状态和 HMI 交互

这个拆分的重点不是“文件看起来整齐”，而是让每个对象有明确边界。

FB_AlarmPoint 不应该知道 HMI 页面怎么画； FB_AlarmManager 不应该直接判断压力是否越限； GVL_AlarmConfig 不应该存运行时确认状态； PRG_AlarmTask 不应该把所有报警点逻辑硬编码在一大片 IF 里。

验证要怎么做

报警框架的验证，不只是“触发一次，看 HMI 亮了没有”。

至少要覆盖这些用例：

1. 条件触发，报警进入 ActiveUnacked。 2. 条件仍存在时点击确认，进入 ActiveAcked。 3. 条件恢复但未确认，进入 RecoveredUnacked。 4. 条件恢复且已确认，回到 Normal。 5. 抑制条件有效时，报警不进入正常活动列表，但保留必要诊断。 6. 同一报警重复触发时，历史记录规则一致。 7. HMI 未确认数量与程序内部状态一致。 8. 掉电或任务重启后，关键状态是否需要保持，规则明确。

这些用例跑通以后，才能说报警框架具备工程复用价值。

这篇文章后续可以怎么变成系列

如果只写一篇 3000 字文章，最适合先讲模型和边界。不要急着塞完整源码，否则读者会在第一篇就被变量、结构体和功能块淹没。

我建议拆成一个 5 篇小系列：

1. 别把报警写成一堆 IF：先把状态模型定下来 讲状态、确认、恢复、抑制和历史边界。

2. CodeSys 报警点功能块怎么写：从布尔条件到状态机 落到 FB_AlarmPoint 的输入、输出、状态迁移和核心 ST 片段。

3. 报警管理器怎么组织：目录、列表、HMI 和历史记录 讲 FB_AlarmManager、报警目录、当前列表和历史结构。

4. 现场排障怎么接进报警系统：提示语不是文案，是工程路径 讲 Cause、Action、ContextValue 和排障表。

5. 报警框架怎么验收：从单点触发到整机异常复盘 讲测试用例、在线变量、HMI 对账和交付清单。

这套顺序更像工程师真正理解一个框架的顺序：先有模型，再有对象，再有界面，再有历史，最后验收。

结尾

报警程序不是项目里最炫的部分，但它最能暴露工程质量。

一个报警框架如果只是一堆报警位，它会在项目后期不断制造隐性成本：HMI 改一次，变量跟着乱；客户问一次历史，工程师临时查；现场误报一次，大家开始怀疑整套系统。

更稳的做法，是从第一天就把报警当成一个小系统设计： 检测层给事实，状态机给规则，目录给语义，历史给追溯，HMI 给视图，验证给信心。
