# ControlRookie 网站项目规则

> 本文件是 `E:\ai-workspace\projects\controlrookie-website` 的项目级强制规则入口。后续 Codex、OpenCode 或其他 AI 自动化工具维护本网站时，必须先读取并遵守本文件。

## 维护目录

- 当前电脑中，ControlRookie 网站维护唯一目录是 `E:\ai-workspace\projects\controlrookie-website`。
- 禁止再创建或使用 `E:\ai-workspace\controlrookie-pages-publish` 作为维护、发布或中转目录。
- 如发布流程确实需要临时目录，必须创建在当前项目根目录下，并使用清晰中文语义命名。
- 当前 Git 顶层是 `E:\ai-workspace`；提交时只能暂存 `projects/controlrookie-website` 下文件，禁止把上级工作区其他改动带入本项目。

## 工程边界

- `项目规划与TODO.md` 是唯一正式规划、路线图、TODO 和执行记录入口，不得新建重复规划文档。
- 正式工程入口是 `src/app/index.html`，样式是 `src/styles/site.css`，交互是 `src/lib/site.js`，内容数据是 `src/content/site-data.js`。
- 当前阶段保持零依赖静态 HTML/CSS/JS；未经用户同意，不得安装 Astro、Next.js、npm 包或其他新依赖。
- `prototypes/homepage-open-design/` 是当前唯一正式首页高保真原型源；ForAI 原站 HTML 只作视觉标尺，不作为开发底座。
- 文章系统开发阶段只允许本地测试；除非用户明确要求上传，否则不得自动同步到 GitHub 官网。

## 视觉底座

- 网站必须延续已验收的 ForAI 黑白灰视觉底座：白底、细边框、轻字重、克制字号、`#f1f1f1` 浅灰卡片和右上角黑色搜索。
- 禁止无依据改变色彩、字号体系、留白、卡片比例、导航行为和桌面单页段落结构。
- 不使用紫色/蓝紫渐变、装饰性光球、过度营销式 hero、复杂滚动动效或破坏工业自动化个人品牌气质的视觉元素。
- 工程代码必须保留控件级中文注释，方便后续手动精细调整。

## 正文边界

- 导航栏以下属于正文内容区；首页、资源、文章、产品、联系等页面必须共享稳定的正文边界框。
- 正文边界是设计和排版约束，边界必须不可见，不是用户可见装饰；禁止为了表达边界而画出可见框线、描边或边框。
- 正文上边界参考首页照片上边缘；上边界到导航栏的距离必须与下边界到页面底边的距离一致。
- 左边界必须与左上角 `CR` logo 的左边缘左对齐，右边界必须与右上角 `EN` 语言按钮的右边缘右对齐。
- 正文边界只是最大安全范围，不是贴边排版目标；正文内容不能超出边界，但必须在边界内部保留舒适内边距、呼吸感和视觉秩序。
- 不要求每个页面把整个边界区域占满；资源页可以相对铺开，单卡页、经历页和联系页必须根据内容量做居中、留白和节奏控制。
- 资源、文章、产品、联系页的第一个主体卡片应与正文上边界顶端形成稳定关系；单卡页面不得因内容高度不同而居中漂移。

## 中心轴布局

- 全站布局设计必须以页面垂直中心轴作为核心参照，导航栏居中分布就是该规则的基准。
- 每个页面的主体内容布局都必须遵循中心轴稳定原则，不得出现明显向左或向右偏移的视觉重心。
- 可以根据内容需要使用左右分栏、单卡、网格或非满幅留白，但整体组合必须围绕页面中心轴平衡。
- 判断页面排版时，优先检查内容组的视觉中心是否接近页面垂直中心轴，再检查局部对齐和边界约束。

## 卡片规则

- 灰色卡片只有在可以点击进入文章、产品、资源详情、外部链接或联系方式时，才允许 hover 或动画效果。
- 不可点击的灰色卡片必须保持静态，不得上浮、旋转、缩放或制造可点击误导。
- 每个信息类灰色卡片必须包含稳定的三段式层级：序号标题、卡片名称、概述。
- 同一行灰色卡片内部三段内容的垂直起点必须严格对齐；各段之间间距固定。
- 同一行灰色卡片高度由该行内容最多的卡片决定，其他卡片跟随等高，不得出现标题和概述上下错位。

## 页面内容规则

- 首页首屏不得显示 `AI 自动化架构师 / 2026` 这类 kicker；`ControlRookie` 的上边缘必须与个人照片顶端形成稳定对齐。
- 首页首屏三段主文案 `ControlRookie`、`跟得上时代的 AI 自动化架构师。`、`致力于控制、算法、架构、前沿技术和 AI 工具的源码开源分享，帮助自动化工程师大幅提升工程效率。` 之间必须保留更舒展的垂直间距。
- 当前首页主体已经由用户确认满意并固定；除非用户明确要求，不得改动首页首屏、照片、主文案、四张领域卡片和对应布局。
- 首页后的个人经历页在导航中必须命名为 `关于我`，英文为 `About Me`，并作为第一个导航按钮；该页必须用清晰可见的时间线表达 `达姆施达特工业大学 -> Bosch -> Fraunhofer IWES -> 新能源 HIL -> 烟草 -> 冶金传动 -> 控制器厂家`，不得只做普通卡片堆叠，并预留后续能力描述区域。
- 文章页和产品页顶部的 `文章`、`产品` 标题必须与主体卡片形成同一内容组，视觉上贴近并从属于卡片区域，禁止标题悬浮在远离卡片的位置。
- 联系页主文案必须使用 `工控人太苦了，让我们一起告别水深火热`；删除原有说明文案和 `ControlRookie / AI Automation Architect`。
- 联系页主文案不得放在灰色卡片背景中，必须以无卡片底的独立排版处理；联系页不得所有卡片挤在上半屏导致下半屏空白，必须围绕页面垂直中心轴做整体平衡。
- 联系页可以使用项目或本地 Logo 图片作为辅助视觉，但禁止使用四个圆角露白或存在白角伪影的 Logo 位图；优先使用无白角伪影的 SVG 或重新处理后的透明/实底素材，并保持黑白灰底座和二维码可读性。

## 文章系统规则

- 文章系统必须支持大量文章长期增长，禁止在单一文章入口页无限罗列文章卡片；必须按分类、系列、标签、主题或时间等维度进行归纳，并提供清晰子导航。
- 文章列表页必须遵循浅导航原则：用户应能用尽量少的操作从分类、系列、搜索或当前文章定位并打开另一篇文章，禁止依赖反复返回首页或单一总入口。
- 单篇文章页必须提供同系列文章、相关分类、上一篇/下一篇、相关文章或相关资源入口，帮助用户在文章之间快速切换。
- 文章、资源、产品、源码、标准、模板和工具之间如存在引用、参考、依赖或上下游关系，必须用可点击超链接建立互跳路径，并提供清晰返回路径。
- 单篇文章页桌面布局必须采用左侧固定目录、右侧标题和正文的阅读结构；左侧目录不随正文滚动消失，正文阅读区必须保持清晰层级和足够行宽控制。
- 单篇文章页右侧必须提供一键回到顶端和一键到底端按钮，按钮位置应固定、低干扰、可键盘访问。
- 文章正文整体字号必须比首页展示型字号更克制；标题、二级标题、三级标题、正文、引用、列表、代码块之间必须有明显层级差异。
- 代码块必须使用适合代码阅读的等宽字体，并为每个代码块提供复制按钮；复制状态必须有明确反馈。
- 每篇文章末尾必须预留评论和回复区；评论、留言、审核、登录、匿名、反垃圾和静态站兼容策略需要在后续总体方案中统一设计，未定策略前不得随意接入第三方评论服务。
- 文章系统仍需沿用当前 ForAI 黑白灰视觉底座，禁止变成传统博客模板、信息门户模板或重营销落地页风格。

## 验收门禁

- 每次结构性或视觉修改后，必须运行 `scripts/verify_production_ready.py`。
- 每次发布前必须运行 `scripts/verify_production_ready.py`，发布后必须运行 `scripts/verify_live_site.py`。
- 每次修改完成后，必须自动弹出浏览器界面预览正式页面，不得让用户手动打开。
- 视觉验证以桌面浏览器视口为主；移动端适配单独验收，不得反向破坏桌面视觉基线。
- 中文文档、源码注释、规则文件写入后必须保持 UTF-8 编码，无乱码、连续问号或 Unicode replacement character。
