### FrontendDesign 反复失效根因

- 根因 1：`E:\ai-relavant\frontenddesign-env\.env` 中保留了 `OPENAI_API_KEY` 等字段，但值为空；Docker Compose 会把空值注入容器，导致后端实际读取不到 key。
- 根因 2：当前 Codex 可用凭据在 `C:\Users\Lenovo\.codex\auth.json`，模型网关配置在 `C:\Users\Lenovo\.codex\config.toml`，不会自动同步到 FrontendDesign `.env`。
- 根因 3：FrontendDesign 后端默认把 OpenAI 模型映射到 `gpt-5.3-codex`，当前 Codex/chrouter 可用模型是 `gpt-5.5`；key 恢复后仍会因为模型名不支持而失败。
- 历史误判：只检查 `.env` 和普通环境变量会得出“缺少供应商 API Key”的结论，但正确恢复路径是同步当前 Codex key/base_url/model，并补丁后端模型映射。

### FrontendDesign 快速恢复步骤

1. 执行 `scripts/恢复FrontendDesign.ps1`。
2. 脚本读取 `C:\Users\Lenovo\.codex\auth.json` 中的 `OPENAI_API_KEY`，读取 `C:\Users\Lenovo\.codex\config.toml` 中的 `base_url` 和 `model`。
3. 脚本写入 `E:\ai-relavant\frontenddesign-env\.env`。文档和终端输出只允许显示字段是否存在和长度，禁止显示真实 key。
4. 脚本通过 Docker Compose 重建 `frontend-design-backend` 与 `frontend-design-frontend`。
5. 脚本在容器内补丁 `/app/llm.py`，让 `get_openai_api_name()` 优先返回环境变量 `OPENAI_MODEL`。
6. 脚本重启后端容器并等待健康状态。
7. 脚本执行两级真实验证：OpenAI Responses API 最小调用，以及 FrontendDesign 自身 `OpenAIProviderSession` 调用。

### FrontendDesign 恢复注意事项

- 禁止把 `E:\ai-relavant\frontenddesign-env\.env` 加入本项目 Git。
- 禁止在日志、文档、提交信息和对话中输出真实 key、token 或完整鉴权头。
- 如果 Codex 后续切换 provider、base_url 或 model，只需要重跑 `scripts/恢复FrontendDesign.ps1`。
- 如果 FrontendDesign 镜像升级后 `/app/llm.py` 结构变化，先重新定位 `get_openai_api_name()`，再调整脚本补丁；不要只改容器临时文件后交付。

## Open Design

- daemon 地址：`http://127.0.0.1:7456`
- 当前要求：必须能创建项目、写入 HTML artifact，并返回可预览 URL。
- 当前状态：daemon 与 MCP artifact 写入链路可用。

### Open Design 根因与修复记录

- 安装路径：`E:\99 装机软件\Open Design`。
- 失败现象：MCP 报错 `cannot reach the Open Design daemon at http://127.0.0.1:7456`。
- 根因 1：桌面进程存在不代表 HTTP daemon 监听 `7456`；当时只有 `opencode` 在 `127.0.0.1:14959` 监听。
- 根因 2：直接用系统 Node.js 22 运行 `daemon-cli.mjs` 会触发 `better-sqlite3` 原生模块 ABI 不匹配，错误为 `NODE_MODULE_VERSION 145` 与 `127` 不一致。
- 修复方式：用 Open Design 自带 Electron Node 运行时启动 daemon，即设置 `ELECTRON_RUN_AS_NODE=1` 后执行 `Open Design.exe ... daemon-cli.mjs --port 7456 --host 127.0.0.1 --no-open`。
- 固化脚本：`scripts/启动OpenDesignDaemon.ps1`。

### Open Design 验证记录

- `127.0.0.1:7456` 已监听，拥有进程为 `Open Design.exe`。
- MCP `create_project` 创建项目 `controlrookie-tool-check` 成功。
- MCP `create_artifact` 写入 `tool-check/index.html` 成功，artifact manifest 状态为 `complete`。

## 验收标准

- 容器或服务启动成功。
- 健康接口通过。
- 能执行真实业务链路，不只检查端口。
- 失败时记录根因、日志和修复动作。
