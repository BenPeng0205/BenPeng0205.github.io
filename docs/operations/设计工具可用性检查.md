# 设计工具可用性检查

## 检查目标

确保 Open Design 与 FrontendDesign 都能用于后续高保真原型设计。

## FrontendDesign

- 前端地址：`http://127.0.0.1:18080`
- 后端健康检查：`http://127.0.0.1:17001/health`
- Docker Compose 路径：`E:\ai-relavant\frontenddesign-env\docker-compose.yml`
- 当前状态：基础服务可用，模型生成链路缺少真实供应商 API Key。

### FrontendDesign 验证记录

- `docker compose -f E:\ai-relavant\frontenddesign-env\docker-compose.yml ps --format json`：`frontend-design-backend` 与 `frontend-design-frontend` 均为 `running` 且 `healthy`。
- `GET http://127.0.0.1:17001/health`：返回 `200`，内容为 `{"status":"ok","service":"frontend-design-backend"}`。
- `GET http://127.0.0.1:18080`：返回 `200`，前端 HTML 可访问。
- `GET http://127.0.0.1:17001/api/design-systems`：返回 `200`，内容为 `[]`。
- 模型生成限制：`.env`、用户级环境变量和机器级环境变量中未发现 `OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`GEMINI_API_KEY`、`REPLICATE_API_KEY`。后端历史日志存在 `No API key`，因此生成能力必须等用户提供真实供应商 API Key 后再复验。

## Open Design

- daemon 地址：`http://127.0.0.1:7456`
- 当前要求：必须能创建项目、写入 HTML artifact，并返回可预览 URL。
- 当前状态：daemon 与 MCP artifact 写入链路可用。

### Open Design 根因与修复记录

- 安装路径：`E:\99 装机软件\Open Design`。
- 失败现象：MCP 报错 `cannot reach the Open Design daemon at http://127.0.0.1:7456`。
- 根因 1：桌面进程存在不代表 HTTP daemon 监听 `7456`；当时只有 `opencode` 在 `127.0.0.1:14959` 监听。
- 根因 2：直接用系统 Node.js 22 运行 `daemon-cli.mjs` 会触发 `better-sqlite3` 原生模块 ABI 不匹配，错误为 `NODE_MODULE_VERSION 145` 与 `127` 不一致。
- 修复方式：用 Open Design 自带 Electron Node 运行时启动 daemon，即设置 `ELECTRON_RUN_AS_NODE=1` 后执行 `Open Design.exe ... daemon-cli.mjs --port 7456 --host 127.0.0.1 --no-open`。
- 固化脚本：`scripts/启动OpenDesignDaemon.ps1`。

### Open Design 验证记录

- `127.0.0.1:7456` 已监听，拥有进程为 `Open Design.exe`。
- MCP `create_project` 创建项目 `controlrookie-tool-check` 成功。
- MCP `create_artifact` 写入 `tool-check/index.html` 成功，artifact manifest 状态为 `complete`。

## 验收标准

- 容器或服务启动成功。
- 健康接口通过。
- 能执行真实业务链路，不只检查端口。
- 失败时记录根因、日志和修复动作。

## 后续复验要求

- 每次正式高保真原型开工前，先运行 `scripts/启动OpenDesignDaemon.ps1`，再用 Open Design MCP 写入一个临时 HTML artifact。
- FrontendDesign 如果要执行模型生成，必须先配置真实供应商 API Key，再用真实生成请求复验，不能把健康检查等同于生成可用。
