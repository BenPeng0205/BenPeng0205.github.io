# 技术路线与工具链

## 最终路线

ControlRookie 网站采用“Open Design 高保真原型先行，再进入正式前端工程”的路线。

## 工具分工

| 工具 | 定位 | 使用边界 |
| --- | --- | --- |
| Open Design | 主设计工具 | 首页、作品墙、核心页面高保真重构与迭代。 |
| ForAI 原站 HTML | 视觉标尺 | 只用于理解字体、间距、结构和截图对照，不作为正式开发底座。 |
| FrontendDesign | 辅助工具 | 后续可用于产品区、博客区、联系区等内容密集模块的备选探索。 |
| Chromium/Playwright | 验收工具 | 辅助桌面浏览器验证和桌面截图，检查首屏、导航、搜索与主视觉可见性。 |
| Git | 版本管理 | 每轮通过验收后提交，过程产物不长期保留。 |

## 为什么不直接改原站 HTML

ForAI 原站 HTML 是 Framer 编译产物，短期像素相似度高，但不适合作为正式项目底座：

- 类名和结构由 Framer 自动生成，不利于长期维护。
- 依赖远程 Framer 脚本、字体、模块和动画状态。
- 修改内容体系、移动端和产品板块时成本高。
- 后续组件化、SEO、博客、产品数据结构都会受限。

## 为什么选择 Open Design

- 输出更接近可维护 HTML/CSS。
- 便于逐屏校准 ForAI 的视觉体系。
- 便于后续迁移到正式前端工程。
- 可控制动画复杂度，符合静态高保真加简单动效的维护目标。

## 后续验收规则

1. 每次首页视觉调整后必须用桌面浏览器实际打开页面，优先 Edge。
2. 需要截图时只生成桌面端截图作为辅助证据。
3. 检查照片占位可见、PLC/AI/Code/Share 一屏内可见、标题不裁切、导航不溢出、搜索可展开并返回匹配结果。
4. 中文文档写入后必须做 UTF-8 乱码检查。
5. 移动端验收放到正式响应式阶段单独执行，不作为当前高保真首页主验收依据。
