MQTT 跑在 TCP 上。PLC 里写 MQTT，真正容易出问题的不是协议名，而是缓冲区、长度字段和字符串拷贝。

  • 适合想理解 MQTT 字符串如何变成 byte buffer 的工程师。

这一层负责什么

应用字符串先进入 AppendString / AppendPayload，再写入 MQTT 报文缓冲区，随后编码 Remaining Length，最后交给 NBS TCP_Write。接收方向则相反：TCP_Read 得到字节，先判断 Remaining Length 和完整帧，再交给报文分发。

载荷追加的真实代码

iecst
METHOD M_AppendPayload : UINT
VAR_INPUT
    sPayload    : STRING(2048);
    pBuffer     : POINTER TO BYTE;
END_VAR
VAR
    uiLen       : UINT;
    i           : DINT;
    pWrite      : POINTER TO BYTE;
END_VAR

// === IMPLEMENTATION ===
uiLen := M_GetStringLength(
    sValue := sPayload,
    uiMaxLen := GVL_Mqtt.cnMaxPayloadSize);

pWrite := pBuffer;
IF uiLen > 0 THEN
    FOR i := 1 TO TO_DINT(uiLen) DO
        pWrite^ := TO_BYTE(sPayload[i - 1]);
        pWrite := pWrite + 1;
    END_FOR
END_IF

M_AppendPayload := uiLen;

这里有两个工程重点：第一，不能让字符串长度超过本地容量；第二，MQTT 报文最终是 byte array，不是 ST 字符串。

缓冲区不是越大越好

发送和接收缓冲区是 4096，默认 payload/topic 是 2048。这是一种平衡：能承载常见 JSON，同时不把 PLC 内存浪费在每个短字段上。

参数影响
cnMaxPayloadSizeST 字符串和 payload 上限
cnSendBufferSize发送完整 MQTT 报文上限
cnRecvBufferSize接收完整 MQTT 报文上限
Broker Maximum Packet Size对端是否允许这么大的报文

这一篇最该记住的几句话

TCP 收到字节不等于 MQTT 收到消息；必须先过 Remaining Length 和完整帧判断。

本篇结论

TCP 层和基础编解码是 MQTT Client 的地基。这里没处理好，后面的 CONNECT、PUBLISH、SUBSCRIBE 会表现成偶发超时、偶发截断、偶发卡状态。

上一篇 / 第12篇 FB_MqttClient 主功能块和状态机怎么跑 MQTT Client 的核心不是 CONNECT 报文，而是状态机。报文只是状态机在某个状态下要做的动作。 下一篇 / 第14篇 CONNECT、PING、DISCONNECT 会话生命周期 MQTT 会话不是 TCP 连接。TCP 连上只是门开了，CONNECT / CONNACK 成功才算进屋。
