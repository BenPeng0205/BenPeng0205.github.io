源码范围说明

本篇属于 MQTT 源码加更，展示的是 V1.0 运行链路中的核心源码片段、对象职责和阅读顺序；它不是完整工程源码包，也不等同于 1.0 全量源码清单。完整源码资源后续应在资源/源码页统一维护。

这一篇不讲报文细节。先解决一个更实际的问题：拿到一套 PLC MQTT Client 源码，第一眼应该看哪里。

适合谁收藏

  • 适合第一次打开 MqttClient_V1_0 源码，不知道从哪里读起的工程师。
  • 适合想先建立源码地图，再看状态机和报文实现的人。
  • 不适合直接查某个 MQTT 报文字节细节，后面几篇会单独展开。

本篇核心图

读图重点：先看对象之间的职责边界，再看状态或数据如何流动。

先给结论

读这套源码不要从 PUBLISH 或 CONNECT 某个方法直接跳进去。正确顺序是：PRG_Test / PLC_PRG -> FB_MqttClient -> GVL_Mqtt -> DUT 数据模型 -> 发送报文方法 -> 接收报文方法 -> TCP 与辅助工具。

原因很简单：MQTT 报文只是结果，真正决定工程能不能稳定跑的是容量、状态、事务和诊断模型。

源码入口

示例程序先实例化 FB_MqttClient，再把连接、发布、订阅、接收变量全部接出来。这个入口对用户最友好，因为它回答的是怎么用。

iecst
PROGRAM PRG_Test
VAR
    bInit                    : BOOL := TRUE;                                                    // 初始化执行标志
    fbMqttClient            : FB_MqttClient;                                                    // MQTT 客户端实例
    bEnable                  : BOOL := TRUE;                                                    // 使能客户端
    bConnect                 : BOOL;                                                            // 连接命令
    sBrokerIP                : STRING(128) := '192.168.20.100';                          // Broker IP 地址
    uiPort                   : UINT := 1883;                                                   // Broker 端口号
    sClientID                : STRING(128) := 'CodeSys_PLC';                             // 客户端标识符
    sUsername                : STRING(128) := '';                                        // 用户名
    sPassword                : STRING(128) := '';                                        // 密码
    eVersion                 : E_MqttVersion := E_MqttVersion.byMqttVersion311;                // MQTT 协议版本
    bCleanSession            : BOOL := TRUE;                                                   // 清理会话标志
    uiKeepAlive              : UINT := 60;                                                     // 心跳周期（秒）
    bUseSSL                  : BOOL := FALSE;                                                  // 是否启用 SSL
    bAutoReconnect           : BOOL := TRUE;                                                   // 是否自动重连
    uiReconnectDelay         : UINT := 5000;                                                   // 重连延时（毫秒）
    udiSessionExpiry         : UDINT := 0;                                                     // 会话过期间隔
    uiReceiveMax             : UINT := 65535;                                                  // 最大接收数量
    udMaxPacketSize          : UDINT := 4096;                                                  // 最大报文长度
    bWillFlag                : BOOL := TRUE;                                                   // 是否启用遗嘱消息
    sWillTopic               : STRING(2048) := 'CodeSys';                                      // 遗嘱主题
    sWillMessage             : STRING(2048) := 'CodeSys Offline';                              // 遗嘱消息内容
    eWillQoS                 : E_MqttQoS := E_MqttQoS.byQoS1;                                 // 遗嘱消息 QoS

这里能看出一个设计原则：身份类字符串保持短，消息类字符串单独放大。这样不会为了一个 IP 或用户名浪费几 KB 内存，也不会让 JSON payload 被默认 STRING 截断。

全局常量决定库的边界

GVL_Mqtt 是这套库的资源预算表。缓冲区、订阅数量、在途消息数量、超时和协议常量都集中在这里。

iecst
{attribute 'qualified_only'}
VAR_GLOBAL CONSTANT

    cnRecvBufferSize              : UINT   := 4096;        // 接收缓冲区大小
    cnMaxSubscriptions            : UINT   := 32;          // 最大订阅数量

    cnMaxTopicAlias               : UINT   := 16;          // 最大主题别名数量
    cnMaxHistory                  : UINT   := 50;          // 最大历史消息数量
    cnMaxPropertyLoop             : UINT   := 100;         // 属性解析最大循环次数
    cnNetworkTimeout              : TIME   := T#10S;       // 网络超时时间
    cnConnectTimeout              : TIME   := T#30S;       // 连接超时时间
    cnResponseTimeout             : TIME   := T#5S;        // 响应超时时间
    cnPublishTimeout              : TIME   := T#10S;       // 发布超时时间
    cnInflightTimeout             : TIME   := T#10S;       // 在途消息超时时间
    cnMaxRetries                  : UINT   := 5;           // 最大重试次数
    cnRetryInterval               : TIME   := T#3S;        // 重试间隔
    cnReconnectBase               : TIME   := T#1S;        // 基础重连间隔
    cnReconnectMax                : TIME   := T#60S;       // 最大重连间隔
    cnWildcardMulti               : STRING := '#';         // 多层通配符
    cnWildcardSingle              : STRING := '+';         // 单层通配符
    cnTopicSeparator              : STRING := '/';         // 主题分隔符
    cnProtocolName                : STRING := 'MQTT';      // 协议名称
    cnProtocolLevelV311           : BYTE   := 4;           // MQTT 3.1.1 协议级别
    cnProtocolLevelV50            : BYTE   := 5;           // MQTT 5.0 协议级别
    cnProtocolLevel               : BYTE   := 5;           // 默认协议级别
    cnDefaultPort                 : UINT   := 1883;        // 默认非加密端口
    cnDefaultPortTls              : UINT   := 8883;        // 默认 TLS 端口
    cnMaxPacketSize               : UDINT  := 268435455;   // MQTT 最大报文长度
    cnMaxPayloadSize              : UINT   := 2048;        // 默认最大载荷长度
    cnMaxTopicLen                 : UINT   := 2048;        // 主题最大长度
    cnMaxClientIdLen              : UINT   := 256;         // ClientId 最大长度

这比把数字散落在每个方法里好维护。现场如果要扩 payload，就看 cnMaxPayloadSize 和缓冲区容量是否匹配；如果要提高高频 QoS1 能力，就看 cnMaxInflight、超时和重试策略。

数据模型要先读

ST_MqttMessage 把收到一条 MQTT 消息抽象成主题、载荷、QoS、Retain、PacketId 和 MQTT 5.0 扩展字段。

iecst
TYPE ST_MqttMessage :
STRUCT
    sTopic               : STRING(2048);  // 消息主题
    sPayload             : STRING(2048);  // 消息载荷文本
    uiPayloadLen         : UINT;          // 载荷长度
    eQoS                 : E_MqttQoS;     // 消息 QoS 等级
    bRetain              : BOOL;          // 保留消息标志
    bDup                 : BOOL;          // 重发标志
    uiMessageId          : UINT;          // 报文标识符
    dtTimestamp          : DT;            // 消息时间戳
    byPayloadFormat      : BYTE;          // 载荷格式指示符
    uiMessageExpiry      : UINT;          // 消息过期间隔（秒）
    sContentType         : STRING(2048);  // 内容类型
    sResponseTopic       : STRING(2048);  // 响应主题
    sCorrelationData     : STRING(2048);  // 关联数据
    uiTopicAlias         : UINT;          // 主题别名值
    bUseTopicAlias       : BOOL;          // 是否使用主题别名
END_STRUCT

这就是从协议到代码的第一层抽象：协议里散落在报文各处的信息，到了工程里要变成结构体字段。

本篇结论

第一篇只需要记住三件事：PRG_Test / PLC_PRG 是用户怎么调用，FB_MqttClient 是运行入口，GVL_Mqtt 和 DUT 是容量、状态、消息、事务的数据模型。下一篇进入主功能块和状态机。
