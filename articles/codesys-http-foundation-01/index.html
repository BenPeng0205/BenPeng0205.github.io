TCP 连接成功、端口可以访问，只能证明字节通道成立。PLC 要稳定处理 HTTP，还必须把消息边界、扫描周期、连接生命周期和诊断证据一起设计。
基础认知，第 1/2 篇；主系列第 01/24 篇。

现场问题

现场最常见的误判，是通信工具显示 Connected，PLC 也拿到了若干字节，于是就把功能判定为已经跑通。真正接入浏览器、curl 或上位机后，问题才集中出现：Header 被拆成两次读取，Body 晚几个周期到达，响应没有明确长度，对端一直等待。

这不是 HTTP 语法知识不足，而是把 TCP 字节流和 HTTP 消息混成了一个对象。PLC 的周期执行模型会把这个错误放大，因为一次事务天然跨越多个扫描周期。

先给结论

本系列只把完整且可验证的 HTTP 事务交给应用层。NBS 负责 TCP，协议内核负责边界，Server 和 Client 负责各自生命周期，业务程序只处理已经完成的请求或响应。

这张图只压缩本篇的判断路径。读图时先找输入边界，再看状态怎样推进，最后确认哪个输出能够作为验收证据。

把对象和边界分开

对象或阶段工程职责现场观察点
TCP/NBS连接、监听、Read、Write、Close句柄、ready、NBS error
HTTP 内核start-line、Header、Body 边界解析状态、长度、协议错误
角色状态机Server 接入或 Client 主动请求状态、超时、连接策略
应用层消费完整请求或响应业务结果和计数

代码为什么这样组织

工程里把最大报文、Header、Body、连接槽位、读写分片和超时全部做成显式常量。它们不是随手填写的数字，而是协议栈的资源边界。以当前实现为例，消息上限 4095 字节、Header 上限 2047 字节、Body 上限 1023 字节、Server 最多 4 个连接槽位。

这些边界决定了教程的结论不能无限外推。本系列讨论明文 HTTP/1.1、Content-Length、入站 chunked、短连接和受控顺序 keep-alive；不把 HTTPS、认证、HTTP/2、pipeline 或连接池包装成已经实现的能力。

下面是与本篇直接相关的真实 ST 连续片段。代码保持源文件原文，不做重写，不用伪代码替代。

iecst
VAR_GLOBAL CONSTANT
    cnDefaultServerPort   : UINT := 8088;                  // 全局协议常量。
    cnMaxClientSlots      : UINT := 4;                     // 全局协议常量。
    cnRxBufferSize        : UINT := 4095;                  // 全局协议常量。
    cnTxBufferSize        : UINT := 4095;                  // 全局协议常量。
    cnMaxMessageSize      : UINT := 4095;                  // 全局协议常量。
    cnMaxHeaderSize       : UINT := 2047;                  // 全局协议常量。
    cnMaxBodySize         : UINT := 1023;                  // 全局协议常量。
    cnMaxTargetLen        : UINT := 255;                   // 全局协议常量。
    cnMaxHostLen          : UINT := 128;                   // 全局协议常量。
    cnMaxHeaderValueLen   : UINT := 255;                   // 全局协议常量。
    cnMaxChunkLineLen     : UINT := 32;                    // 全局协议常量。
    cnTcpConnectTimeoutUs : UDINT := 5000000;              // 全局协议常量。
    cnClientTimeoutMs     : UDINT := 8000;                 // 全局协议常量。
    cnServerConnectionMs  : UDINT := 8000;                 // 全局协议常量。
    cnWriteTimeout        : TIME := T#5S;                  // 全局协议常量。
    cnTcpReadChunkSize    : UINT := 240;                   // NBS 单次 TCP_Read 字节数，避开部分运行时 255 字节边界。
    cnTcpWriteChunkSize   : UINT := 240;                   // NBS 单次 TCP_Write 字节数，大响应按 TCP 字节流分批发送。
    cnTcpReadMaxErrorCount: UINT := 3;                     // TCP_Read 连续错误滤波次数，避免大报文半包接收被瞬时读错误误杀。
    cnDefaultContentType  : STRING(32) := 'text/plain';    // 全局协议常量。
    cnDefaultPath         : STRING(8) := '/';              // 全局协议常量。
END_VAR

源码阅读时不要只找某个 TRUE。先确认输入是否被锁存，再看状态和长度怎样变化，最后检查错误出口是否保留了足够诊断信息。

验证路径

场景操作通过口径
字节层确认监听句柄或主动连接句柄有效不能据此宣布 HTTP 成功
协议层确认 Header 和 Body 边界完整状态机才能进入完成态
外部端看到合法状态码、Header 和 Body证明对端认可报文
PLC 在线量状态、错误、计数和快照一致证明工程侧可诊断

验证结论分为三层：源码能够证明实现意图，离线用例能够证明确定输入下的边界行为，真实通信能够证明目标运行时与外部工具之间确实闭环。三层证据不能互相替代。

常见误判

  • 把端口可连接写成 HTTP 协议栈已经通过，忽略消息边界和响应证据。
  • 只在外部工具看结果，不核对 PLC 状态、错误码和计数是否同步变化。
  • 把当前 4095 字节消息上限当成 HTTP 标准限制，而不是本工程资源取舍。

本篇结论

  • 连接成功不等于 HTTP 成功。
  • PLC 必须跨周期累积并判断消息边界。
  • 任何发布结论都要同时给出外部证据和 PLC 在线证据。
