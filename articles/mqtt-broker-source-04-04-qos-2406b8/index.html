Broker 高频通信能不能稳，关键不在能不能转发，而在 ACK、重发、队列和慢客户端保护。

调度链路

路由投递任务进入连接投递队列，连接槽位通过 PrepareNextTx 组帧，TCP_Write 批量写出。QoS1/QoS2 消息进入 TxScheduler 等 ACK；超时则重发并设置 DUP。

出站事务登记

METHOD M_RegisterPublish : BOOL
VAR_INPUT
    stPublish : ST_MqttBrokerPublishFrame; // 已发送或即将发送给订阅者的发布帧
    udiNowMs  : ULINT; // 当前系统时间戳[ms]
END_VAR
VAR
    uiIndex   : UINT; // 出站事务表扫描索引
    uiFree    : UINT; // 首个空闲出站事务槽位索引
END_VAR

// === IMPLEMENTATION ===
IF stPublish.eQoS = E_MqttQoS.byQoS0 THEN
    M_RegisterPublish := TRUE;
    RETURN;
END_IF

IF stPublish.uiPacketId = 0 THEN
    eLastError := E_MqttBrokerError.uiProtocolMalformed;
    M_RegisterPublish := FALSE;
    RETURN;
END_IF

QoS0 不进入事务表，QoS1/QoS2 必须有 PacketId。没有 PacketId 的 QoS1 发布，本身就是协议错误。

事务表保存的是这条消息发给哪个客户端、PacketId 是多少、当前等哪个 ACK、什么时候重发。没有这张表，Broker 在高频 QoS1 场景下只能靠猜。

诊断闭环

Broker 的诊断比 Client 更重要，因为问题可能出在任意一个客户端、任意一个槽位、任意一条订阅或任意一个 ACK。连接快照、诊断历史、统计计数和队列水位，就是为了在线定位这些问题。

这一篇最该记住的几句话

Broker 高频稳定性的核心是队列、ACK、重发和诊断，不是简单收到 PUBLISH 就转发。

本篇结论

Broker 源码最后要看的不是某个报文，而是调度闭环：投递任务进入队列，队列批量写出，QoS 事务等待 ACK，超时重发，诊断变量说明发生了什么。这套闭环跑稳后，PLC 侧 Broker 才能承受多个客户端同时发布订阅。
